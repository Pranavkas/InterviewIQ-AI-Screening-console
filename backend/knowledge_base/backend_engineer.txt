REST API Design
A well-designed REST API models resources as nouns and uses HTTP verbs (GET, POST, PUT, PATCH, DELETE) to act on them. Good API design favors consistent naming, proper use of status codes, versioning strategies (URI versioning like /v1/, or header-based versioning), pagination for large collections, and idempotency for PUT/DELETE operations. Overly chatty APIs increase latency; overly coarse APIs reduce flexibility, so designers balance resource granularity against client needs. HATEOAS is a maturity level that embeds links to related actions but is rarely fully implemented in practice.

Database Indexing
An index is a data structure (commonly a B-tree or hash structure) that allows the database to find rows without scanning the entire table. Indexes speed up read queries but slow down writes because each insert/update/delete must also update the index. Composite indexes are ordered, so a query must use a left-prefix of the indexed columns to benefit. Over-indexing wastes storage and write throughput; under-indexing causes full table scans. Covering indexes let a query be answered from the index alone without touching the table.

Caching Strategies
Caching reduces load on databases and downstream services. Cache-aside (lazy loading) has the application check the cache first, and populate it on a miss. Write-through caching updates the cache and database together. Write-behind caching updates the cache immediately but writes to the database asynchronously. Cache invalidation is one of the hardest problems in computer science: strategies include TTL expiry, explicit invalidation events, and versioned cache keys. Redis and Memcached are common in-memory caches; CDN caching addresses static asset delivery.

Concurrency and Threading
Concurrency allows a system to make progress on multiple tasks; parallelism executes tasks simultaneously on multiple cores. In Python, the Global Interpreter Lock (GIL) limits true parallel execution of threads for CPU-bound work, so CPU-bound tasks often use multiprocessing while I/O-bound tasks benefit from threading or async/await. Race conditions occur when shared state is modified by multiple threads without synchronization; mutexes, semaphores, and atomic operations are common mitigations. Deadlocks arise when two or more threads wait on each other's locks indefinitely.

System Design and Scalability
Horizontal scaling adds more machines; vertical scaling adds more resources to a single machine. Load balancers distribute traffic across instances using algorithms like round-robin, least-connections, or consistent hashing. Database scaling techniques include read replicas for read-heavy workloads and sharding for write-heavy workloads, partitioning data by a shard key. The CAP theorem states a distributed system can only guarantee two of Consistency, Availability, and Partition tolerance during a network partition. Rate limiting (token bucket, leaky bucket) protects services from being overwhelmed.

Authentication and Authorization
Authentication verifies who a user is; authorization determines what they can do. Common patterns include session-based auth with cookies, and token-based auth using JWTs, which are self-contained and stateless but harder to revoke before expiry. OAuth2 is a delegation protocol allowing third-party access without sharing credentials, commonly paired with OpenID Connect for identity. Role-based access control (RBAC) assigns permissions via roles, while attribute-based access control (ABAC) evaluates policies against user/resource attributes for finer-grained control.

Message Queues
Message queues like RabbitMQ and Kafka decouple producers from consumers, enabling asynchronous processing and smoothing out traffic spikes. Kafka is a distributed log optimized for high-throughput streaming and replay, using partitions for parallelism and consumer groups for scaling consumption. RabbitMQ is a traditional broker supporting flexible routing (direct, topic, fanout exchanges) and is well-suited for task queues. At-least-once delivery requires idempotent consumers to avoid duplicate processing; exactly-once semantics are hard to guarantee across distributed systems.
